## High-Confidence Findings

### 1. File-too-large errors on first Read attempt are a systemic pattern

**Confidence:** High — observed in 5+ runs across all workflow types

Files like `docs/development/roadmap.md` (10,645 tokens), `docs/standards/workflow-scripts.md` (11,063 tokens), and log files (28k–80k tokens) consistently exceed the Read tool's token limit on first attempt. The agent always recovers (using offset/limit or switching to Grep/jq), but each failed read wastes a turn and tokens for the error response.

**Evidence:**
- `revision-major-105659` line 5: `roadmap.md` exceeds token limit
- `revision-major-115958` lines 5, 7: `roadmap.md` and `workflow-scripts.md` both exceed limits
- `revision-major-115958` lines 194, 200: subagent task output (28,910 tokens) exceeds limit
- `review-runs-120344` lines 11, 13, 17, 31, 41, 43: six file-too-large errors on log files
- `review-runs-120558` line 10: revision-major log at 84,095 tokens

**Recommendation:** Add guidance to workflow prompts instructing agents to use `limit: 200` on first read of standards/roadmap documents, and to use Grep or `wc -l` before attempting to read any log file. This is a prompt-level change, not a script change.

**Impact:** Saves 1–6 wasted tool calls per run depending on workflow type.

### 2. Rate limit utilization runs hot during back-to-back workflows

**Confidence:** High — observed in 3+ runs within the same 5-hour window

Multi-stage workflows (build-phase, revision-major) consistently push token bucket utilization to 93–99% when run consecutively. The system has not yet hit an actual rate limit stall, but it is one moderately-sized run away from one.

**Evidence:**
- `revision-major-124205` line 132: `"status":"allowed_warning","utilization":0.99` (8 warning events)
- `build-phase-123551` line 183: `"utilization":0.93` (4 warning events)
- `revision-major-121901`: rate limit warnings present
- `build-phase-143445`: no warnings (ran in a separate window, confirming the pattern is about consecutive runs)

**Recommendation:** Add a pre-run rate limit check to the workflow shell scripts. A simple utilization query before invoking Claude would allow the script to delay or warn rather than risk a mid-workflow stall.

**Impact:** Prevents potential mid-workflow stalls that would waste all progress in a partially-completed run.

### 3. Sensitive path guard blocks test fixture creation in worktrees

**Confidence:** High — 5 consecutive permission denials in a single run, confirmed in result record

When agents try to create test data inside a worktree's `.claude/logs/` path, the permission system treats it as a sensitive file operation and blocks it. This caused a 5-turn retry loop in the `revision-major-115958` run before the agent found a workaround.

**Evidence:**
- `revision-major-115958` lines 59, 61, 63, 65, 67: five consecutive denied Bash commands
- `revision-major-115958` line 255: `permission_denials` array explicitly lists all 5 denied commands
- Agent recovered by falling back to `bash -n` syntax checking

**Recommendation:** Add a note to `docs/standards/workflow-scripts.md` that test fixtures for scripts reading `.claude/logs/` must be placed in a non-`.claude` path (e.g., `/tmp/` or `tests/fixtures/`).

**Impact:** Recovers ~5 turns per run of any script involving log-reading test cases.

## Medium-Confidence Findings

### 6. Parallel Bash cancellations inflate error counts

**Confidence:** Medium — observed in 3 runs, consistent mechanism

When two Bash calls run in parallel and one fails, the other is cancelled with `is_error: true`. These are not real errors but inflate raw error counts. Actual unexpected errors across all 12 runs are very low (estimated 4–6 total, vs. 20+ raw `is_error` events).

**Evidence:** `revision-major-121028` lines 65-66, 170-171; `revision-major-115958` lines 91, 102

**Recommendation:** When analyzing logs, filter out `Cancelled: parallel tool call` entries. No workflow change needed.

**Needs:** Useful context for future review-runs analyses.
